CBDDS Workflow Overview

CBDDS is currently used as a DDS4CCM design tool and deployment planning tool.

Designing Components and Applications

CBDDS supports both "top-down" and "bottom-up" workflows.

In top-down, architects decompose the system into its constituent components and design their connections by starting with the highest level component assemblies and decomposing each component into another assembly or a monolithic component. The component interfaces are then defined in detail.

Bottom-up design starts with the specification of the interfaces and messages. These interfaces may be modeled using CBDDS or imported from externally created IDL. Components are then designed with ports that implement these interfaces. Architects can then assemble these well-defined components into higher-level components or applications.

CBDDS in non-prescriptive as to the workflow a development team may use. Combinations of top-down and bottom-up can be used by multi-person, multi-discipline, multi-location teams.

Team members can validate their designs at any point during the workflow.

Upon completion of a design workflow, CBDDS will generate the IDL 2, 3, and 3+ files that define the interfaces and components.

Deployment Planning

CBDDS is used to

  • Model one or more target environments

  • Create deployment plans

  • Generate deployment plans

A target environment or Domain is comprised of the systems computing nodes, interconnects, and bridges. CBDDS is used to model a target (what nodes and how they are connected) and specify the resources provided and deployment planning requirements.

A deployment plan consists of a specific assignment of software components to a target environment. This also includes assignment of components to processes for collocation or separation.

Once a valid deployment plan has been created, CBDDS can generate the deployment artefacts required by the DAnCE deployment management software. This includes the CDD file for the target and the CDP file for the deployment.
